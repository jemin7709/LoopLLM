# AI Agent Guidelines (AGENTS.md)

이 문서는 프로젝트 내에서 활동하는 모든 AI 코드 에이전트 및 어시스턴트(GitHub Copilot 등)가 코드를 작성하고 수정할 때 지켜야 할 핵심 지침입니다.

## 핵심 코딩 원칙

> **대원칙 (Golden Rule):** 이하의 어떤 원칙이든 완벽하게 지키려는 시도가 코드를 불필요하게 복잡하게 만든다면, 항상 **KISS 원칙**과 **실용성**을 최우선으로 판단하여 타협점을 찾으세요.

### 1. KISS (Keep It Simple, Stupid)
- 가장 단순하고 기능적인 해결책을 최우선으로 선택합니다.
- 코드의 목적이 명확하게 드러나야 하며, 누구나 직관적으로 읽고 이해할 수 있도록 작성하세요.

### 2. YAGNI & 오버엔지니어링 금지 (No Overengineering)
- 현재의 요구사항을 확실히 충족하는 데 필요한 만큼만 코드를 작성하세요.
- "나중에 필요할지도 모른다"는 추측을 바탕으로 불필요한 추상화나 복잡한 디자인 패턴을 미리 도입하지 마세요.

### 3. 클린 코드 프랙티스 (Clean Code)
- **명확한 명명**: 변수, 함수, 클래스는 코드가 하는 일을 명확히 보여주도록 구체적으로 네이밍합니다.
- **SRP (단일 책임)**: 함수와 모듈은 단 하나의 명확한 목적만 가지도록 짧고 간결하게 유지합니다.
- **코드 그 자체가 주석**: 동작(What)을 장황하게 설명하는 주석은 피하고, 코드로 표현하세요. 주석에는 코드를 그렇게 짠 배경과 이유(Why)만을 기록합니다.

### 4. 방어적 프로그래밍 지양 (No Defensive Programming)
- 시스템 내부 도처에 발생 가능성이 희박한 예외 처리나 불필요한 `null` 체크를 남발하지 마세요.
- 입력값 검증은 시스템의 '경계'(예: API 진입점)에서만 한 번 확실하게 처리하고, 내부 핵심 로직은 정제된 데이터를 신뢰하여 간결하게 작성합니다.
- 오류가 발생하면 숨기지 말고 즉시 예외를 발생시키는 **Fail Fast** 방식을 통해 버그를 빠르게 노출시킵니다.

### 5. 실용적인 SOLID 적용 (Pragmatic SOLID)
- 설계 시 SOLID 원칙을 참고하여 유지보수성을 높이는 방향을 지향하되, 이 원칙을 맹신하여 기계적으로 적용하지 마세요.

### 6. 테스트 코드 작성 원칙 (Testing)
- 명시적으로 테스트 코드 작성을 요청받았을 때만 단위 테스트나 관련 테스트 코드를 작성하세요.
- 사용자의 지시 없이 선제적(Proactive)으로 테스트 파일을 생성하거나 코드를 추가하지 마세요.
